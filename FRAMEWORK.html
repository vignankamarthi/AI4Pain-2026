1. Why this framework exists

Let's start with the situation we walked into.

The AI4Pain 2026 Grand Challenge asks you to classify each trial of peripheral autonomic signals (BVP, EDA, RESP, SpO2) into one of three classes: No Pain, Arm Pain, Hand Pain. Each subject contributes 36 trials. There are 41 training subjects, 12 validation, 12 test, and test labels are blinded. You get up to 5 emailed CSV submissions, total.

Now pause and look at that channel list again: BVP, EDA, RESP, SpO2. All four are peripheral autonomic measurements. None of them are cortical. And the task is asking you to distinguish arm pain from hand pain — a localisation distinction. The information that tells you whether a person's arm hurts or their hand hurts is largely cortical. Peripheral autonomic signals know about pain, but they don't know much about where.

That's the situation. The benchmark is asking you to extract localisation information from a channel set that mostly doesn't carry it. Published prior work on this kind of task sits in the 50–60% balanced-accuracy range. Random guessing is 33%.

Our thesis from day one: the benchmark is flawed, and any winning model must extract every last bit of signal from a low-information channel set. So we are not going to write one model and tune it. We are going to search. Hard. With every diversity-preserving mechanism we can justify from the literature.

That decision — "we will search hard" — is what produces the framework. The rest of this doc is the answer to one question: what does it actually mean to search hard?

What we want from a search loop

If you were going to build a model selection loop from scratch, what would you want it to do?

  1. Try many genuinely different things. Not "the same model with different learning rates." Different architectures, different signal representations, different inductive biases.
  2. Not get stuck. If the search converges onto one family of solutions, you've lost the diversity benefit. You need something that pushes away from local optima.
  3. Score every candidate honestly. Single-number summaries hide failure modes. Balanced accuracy alone doesn't tell you which class is being starved. So we want a vector of metrics, not a scalar.
  4. Resist gaming. Any single-axis fitness will be gamed. If you optimize raw accuracy, the loop finds models that overfit. If you penalize overfitting too hard, the loop ships toys.
  5. Reject illegal candidates immediately. The challenge has hard rules (no external data, no pretrained weights). A program that violates them is not a "weak" candidate; it should never reach the cluster.
  6. See its own progress. The loop should know when it's stuck, when it's over-rejecting, when it's collapsing into a monoculture. And it should be able to respond.
  7. Be reproducible. Every spec, every fitness vector, every constraint rejection, every state mutation lives in a ledger. The loop must be resumable from any point. The cluster and the laptop must agree on what happened.

That's the wish list. Now the framework. It has three levels because the wish list breaks naturally into three layers of concern: what we evolve (the candidates), how we evolve them (the engine), and how we change the engine itself when it's stuck.

2. The big picture (three levels, one diagram)

Before we open any boxes, here is the whole system. One diagram. Three levels. Read it from the bottom up: programs (Level 0) are produced and consumed by the engine (Level 1), and the engine itself is mutated by the introspection layer (Level 2). Everything to the left is "what mutates," everything to the right is "what triggers a mutation."

  |                                                                 |
  |  LEVEL 2  --  Framework genome mutation                         |
  |                                                                 |
  |  Mutates : every Level 1 parameter and operator choice          |
  |  Trigger : an empirical compound detector over the ledger       |
  |            (stagnation OR over-rejection OR diversity collapse) |
  |  Gates   : bootstrap floor, HIP-H human confirmation            |
  |  Lineage : Schmidhuber Goedel Machines, AlphaEvolve, POET       |
  |                                                                 |
  +---------------------+-------------------------------------------+
                        | reads ledger -> proposes genome diff
                        v
  +-----------------------------------------------------------------+
  |                                                                 |
  |  LEVEL 1  --  The evolutionary engine                           |
  |                                                                 |
  |  Engine   : FunSearch islands + GENITOR replacement             |
  |  Scoring  : Pareto + Novelty + ECE penalty + Failure boost      |
  |  Filters  : Rule guards -> AST tabu -> Lineage cap              |
  |  Breakers : Migration + Coevolutionary critic + Stagnation      |

Figure 1. The whole system. Three levels, two arrows.

A few things to notice before we start pulling layers apart.

Level 0 turns over every iteration. Every step of the loop, one new candidate program is created, trained, scored, and either replaces an existing population member or gets rejected. The population is in constant motion.

Level 1 is held fixed across iterations. Almost. The structure of Level 1 doesn't change — same islands, same GENITOR rule, same constraint pipeline. But Level 1 has a few knobs that drift each step: how often the prompt biases toward literature versus novelty, how aggressive the novelty bonus is, how high the temperature is. Those drifts are the meta-stochastic layer, and they live inside Level 1.

Level 2 is held fixed UNTIL it isn't. Most of the time, Level 2 is silent. It just watches. But when the ledger shows that Level 1 is genuinely stuck — not noisy, stuck — Level 2 proposes a structural change to Level 1 itself. Maybe scale up novelty weight, maybe add an island, maybe swap a replacement rule. That proposal is gated by a human (Vignan), and if approved, Level 1's structure changes for all future iterations.

A note on language

People in the genetic algorithms literature sometimes use "meta-evolution" or "meta-stochastic" to mean either of these layers. We are using a specific split: meta-stochastic = the drifting knobs inside Level 1 (small, per-iteration, no human in the loop), and Level 2 = the introspection layer that structurally mutates Level 1 (rare, threshold-triggered, human-gated). Both are real, but they operate at different cadences and need different mechanisms. Don't conflate them.

OK. That's the whole system at altitude. Now let's start opening boxes, beginning at the bottom.

3. Level 0: what is a "program"?

The thing the loop evolves is a program. So before we explain how the loop picks them, mutates them, or scores them, we need to be clear about what one of them looks like.

A program is not a Python file. A program is a spec: a JSON object describing how to assemble a training pipeline from named components. The framework then renders the spec into a PyTorch+sklearn run-script.

Here is what a real spec from our population looks like, slightly cleaned up for readability.

{
  "family": "multi_stream_bigru",
  "preprocessing": {
    "resample_hz": 100,
    "normalize": "z_score",
    "window_sec": 36
  },
  "feature_extraction": null,
  "model": {
    "type": "MultiStreamNet",
    "encoder_type": "bilstm",
    "hidden": 40,
    "layers": 2,
    "fusion": "late_concat",
    "dropout": 0.3
  },
  "training": {
    "loss": "focal",
    "focal_gamma": 1.0,
    "optimizer": "adamw",
    "lr": 1.0e-3,
    "epochs": 150,
    "batch_size": 8,
    "n_seeds": 3
  },
  "decode": {
    "agg": "argmax"
  },
  "reasoning_summary": "Multi-stream encoder with BiLSTM per channel
   and late concat fusion. Per-seed checkpointing; n_seeds=3 to absorb
   the seed-variance lottery exposed in iter_0014."
}

Five top-level keys describe the entire run: preprocessing, feature_extraction, model, training, decode. Call these the five genes. Plus a family tag and a reasoning_summary (the LLM's own rationale for this child — we'll need this for observability later).

Why split it into five genes specifically? Because we want a clean crossover operator. If you can replace one gene at a time (or pick each gene independently from one of two parents), you get genuine recombination across the population, not just point mutations on a flat dictionary. The genes are the units of inheritance in this evolutionary algorithm.

Vocabulary

We'll use spec and program interchangeably from here on. A family is a class of architecturally similar programs (e.g. multi_stream_bigru, spectrogram_cnn2d, dual_ensemble). A child is a single mutated spec produced from one or two parents. A batch is the set of children evaluated together on the cluster (one per active island per iteration).

From spec to PyTorch

The spec is rendered by framework/render.py. Each family has a registered entry point in FAMILY_ENTRY_POINTS that maps the spec's model.type + parameters to a concrete training script. The entry points live in ai4pain/:

  ---------------------------------------------------------
  multi_stream_bigru     -->  ai4pain/multi_stream.py
  multi_stream_aux       -->  ai4pain/multi_stream.py (aux head)
  spectrogram_cnn2d      -->  ai4pain/cnn_1d.py (2D variant)
  residual_spectrogram   -->  ai4pain/cnn_1d.py (residual variant)
  dual_ensemble          -->  ai4pain/multi_stream.py (joint head)
  hrv_features_mlp       -->  ai4pain/baselines.py
  minirocket             -->  ai4pain/minirocket.py
  ...

Figure 2. Family registry. Adding a new family means adding one row here plus a run_from_dir in the corresponding module.

The rendered script writes its output to experiments/iter_NNNN/child_MM/result.json. That's the contract: the framework hands the cluster a directory with spec.json and run.py; the cluster hands back a result.json with the fitness vector. Nothing more, nothing less.

What does a result.json look like?

The fitness vector is the only thing the loop cares about. Here's the shape:

{
  "balanced_accuracy_3class": 0.5274,
  "balanced_accuracy_3class_std": 0.0071,
  "per_class_precision": [0.61, 0.43, 0.52],
  "per_class_recall":    [0.69, 0.41, 0.48],
  "confusion_matrix": [[99,21,24], [27,59,58], [25,49,70]],
  "macro_f1": 0.519,
  "auc_ovr": 0.71,
  "ece": 0.083,
  "binary": {
    "balanced_accuracy": 0.8194,
    "pain_precision": 0.84, "pain_recall": 0.81,
    "auc": 0.88,
    "confusion_2x2": [[112,32], [55,233]]
  },
  "generalization_gap": 0.038,
  "param_count": 124931,
  "train_wallclock_sec": 412,
  "inference_wallclock_sec": 8,
  "per_seed": {
    "seeds": [0,1,2],
    "balanced_accuracy_3class": [0.5210, 0.5301, 0.5311]
  }
}

Notice what is and isn't in there.

Every metric the loop will ever score against is here — not just the headline number. Balanced accuracy is the headline, but we also need the generalization gap (for Pareto), the ECE (for the confidence penalty), the parameter count (for Pareto), the wallclock (for budget enforcement), and the confusion matrix (for novelty distance in confusion-matrix space). Stripping any of these to "just save the accuracy" would force a re-run when the loop later asks for them.

There's also a per-seed block. This is here because we learned the hard way that single-seed fitness is untrustworthy on this dataset (seed-to-seed variance was around 0.11 absolute on 3-class accuracy — we'll come back to this in Section 4). Every fitness value the loop reads is a mean over n_seeds independent runs, with the per-seed list preserved for diagnostics.

And there's a binary block. The official task is 3-class, but we also project predictions to Pain-vs-No-Pain (collapsing AP and HP into one). This is purely diagnostic — the loop's tournament selection only ever uses 3-class balanced accuracy — but the binary block is the empirical fingerprint of "the autonomic signal carries pain information, just not localisation." We want every result.json to carry that fingerprint so the loop can never lose sight of it.

That's a program. A spec, a rendered script, a result.json with a fitness vector. The population is just a list of these, sorted by fitness, partitioned across islands. Now we can talk about how the loop turns one into another.

4. Level 1, part A: the evolutionary engine

Now we have a population of programs. How do we make the next one?

If you take the simplest possible evolutionary algorithm, here is what one step looks like:

  1. pick a parent from the population
  2. mutate it to produce a child
  3. score the child
  4. insert the child, evict the worst

That's it. Pick, mutate, score, replace. Every fancier algorithm is some elaboration of this loop, and that's what Level 1 is: a careful elaboration of these four steps to do them well on a slow, expensive evaluation budget (each child takes minutes-to-hours of cluster time).

The elaborations come from two sources in the literature: FunSearch (Romera-Paredes et al., Nature 2024) for the population structure, and GENITOR (Whitley 1988) for the replacement rule. Let's take them in that order.

FunSearch: islands, not a global pool

If you keep one big population and always pick parents from it, the population quickly converges. The best program gets copied; near-copies of it dominate. Diversity dies. With one global pool, a single bad evaluation streak can kill global progress, because every parent in the pool descends from the same recent winners.

FunSearch's answer is to keep M islands in parallel, each holding its own subpopulation of K programs. Mutations happen within an island. Once in a while, the worst-performing islands get reset (their bottom-K programs reseed from the global champion plus perturbation), but most of the time the islands evolve independently.

       +-----+   +-----+   +-----+   +-----+
       | I0  |   | I1  |   | I2  |   | I3  |     M islands
       +--+--+   +--+--+   +--+--+   +--+--+     (default M=4)
          |         |         |         |
          v         v         v         v
       K programs per island, rank-sorted by fitness    (default K=8)

  iteration step on island I_j:
    1. rank-based tournament selection -> parent
    2. mutation operator (Claude Code session) reads
       a context blob, writes child spec JSON
    3. constraint pipeline (Section 6) -> reject + regenerate if blocked
    4. render to PyTorch (framework/render.py) -> cluster eval
       -> fitness vector
    5. GENITOR replacement: evict lowest-fitness member, insert child
       island size stays K (steady-state, not generational)

  every R iterations (reset_cadence, default 50):
    bottom-K islands by best-fitness reseed from
    the global champion plus perturbation

Figure 3. One step of the engine, on one island. The whole population is M of these running in parallel.

The key insight is that isolation preserves diversity. A bad mutation in island I2 doesn't poison I0's gene pool. If I2's whole tree of descendants is a dead end, the reset cadence eventually catches it. Meanwhile, I0 has been evolving in a completely different direction.

GENITOR: one child at a time, not a generation

Classical genetic algorithms work in generations: you evaluate the whole population, then produce a whole new population at once. That works fine when each evaluation is fast and parallel.

It doesn't work for us. Each evaluation here is a full PyTorch training run on the cluster. Generations would mean: ship 32 children, wait for all of them to finish, then pick parents for the next generation. The slowest training run blocks the entire population from advancing.

GENITOR (Whitley 1988) is the steady-state alternative: one child per step. You produce one child, score it, evict the worst current member, insert the child. Repeat. No synchronization barriers. Each cluster job can return whenever it's ready and the population just keeps moving.

For us, "one child per step" actually means "one child per island per batch": we ship N children together (one per active island), which the cluster's SLURM array runs in parallel up to a concurrent cap. But the replacement rule is per-island steady-state, not generational. The island's size stays K throughout.

Why "the mutation operator IS this Claude Code session"

In classical FunSearch, the mutation operator is a paid LLM API call. We don't have that budget, and the project's hard rules forbid an external LLM SDK in requirements.txt. So the mutation operator is this Claude Code session. framework/mutation.py doesn't call an API; it assembles a context blob (recent best programs in the parent's island, fitness vectors, meta-stochastic state, constraint feedback from the last few rejections). The session reads that blob conversationally and emits the child spec. No SDK. No API key. The framework is hand-driven, by design.

What does a mutation actually do?

Most generic evolutionary algorithms either do point mutations (perturb one number) or crossover (combine two parents). We do both, but we also do something stronger: architectural operators that move along axes orthogonal to hyperparameters.

Why? Because we lived through iterations 12–14, where the loop did nothing but hyperparameter tweaks, and the entire population stalled inside its own seed-noise floor. Bumping hidden=40 to hidden=56 changes the AST fingerprint, but doesn't change the inductive bias of the model. To produce genuinely different architectures, we need operators that change structure, not numbers.

The four structural operators that landed in the iter_0015 rebuild:

swap_encoder

In the multi_stream family, replace the per-channel encoder. encoder_type is drawn from {gru, lstm, bilstm, transformer, conv1d}. Each gives the network a fundamentally different inductive bias: GRU is fast and recurrent; transformer attends globally with no recurrence; conv1d is local and translation-equivariant. Same surrounding pipeline, different temporal prior.

swap_fusion

Change how the per-channel embeddings combine before the classifier. fusion is drawn from {late_concat, mean_pool, max_pool, attention_pool, mid_fusion}. Different fusion = different way of routing inter-channel information.

add_aux_stream

Branch into a sibling family multi_stream_aux: same MultiStreamNet body, but fused with a 26-dim HRV-feature side-MLP. The network sees both the raw signal AND its autonomic summary. Lets the loop test whether engineered features carry signal the deep encoder is missing.

graft_family

A fully-random per-gene crossover of two parent specs. Each of the 5 genes (preprocessing, feature_extraction, model, training, decode) is independently picked from parent A or parent B at 50/50. predict_graft_coherence tags type-contract violations (e.g. CNN model with non-spectrogram input), but incoherent grafts are tagged, not blocked: the ledger tolerates a missing result.json and Level 2 can read the failure rate. Sometimes the incoherence is the discovery.

These four are the structural teeth of the search. AST tabu (Section 6) catches near-duplicate fingerprints, but it can't catch monoculture by itself: hidden=40 and hidden=56 are different fingerprints. The structural operators force the search to walk across families, not just along hyperparameter axes.

Multi-seed fitness (a lesson learned the hard way)

Earlier we said the result.json carries a per_seed block. Here's the story behind it.

iter_0013 returned a champion with val balanced accuracy 0.5486. Big win, well above the population. iter_0014 reseeded the same configuration with a different random seed, and the result was 0.4352. Same code, same data, same hyperparameters — an absolute spread of ~0.11 between two seeds.

That single fact invalidated every fitness comparison the loop had been making. A 0.5486 "champion" and a 0.4352 "failure" could be the same spec. Tournament selection was picking parents based on noise. The loop was thrashing.

The fix: every fitness evaluation runs n_seeds independent training runs (default 3 during search, 20 at submission time). The fitness vector reports the mean, with the per-seed list preserved. The tournament selection sees the mean. The Pareto sort sees the mean. The novelty distance sees the mean.

This dropped the per-spec sigma from ~0.11 to ~0.007–0.013, which is the floor of what 12 val subjects can resolve. Most importantly, it dropped it below the differences between architecturally distinct specs — so the loop could finally see signal instead of noise.

OK. We now have a parent, a child spec produced by mutation, and a way to replace the worst member of an island. But we still haven't said how we score. That's the next section.

5. Level 1, part B: the scoring stack

Vanilla FunSearch returns a single scalar: "did the program solve the problem better than the previous best." We can't use a single scalar, for a reason that's easy to demonstrate.

Suppose the loop produced two children, both at val balanced accuracy 0.52:

  • Child A: 0.52 val, 0.95 train. Generalization gap = 0.43. Memorized the training set.
  • Child B: 0.52 val, 0.54 train. Generalization gap = 0.02. Honest.

By raw balanced accuracy, A and B tie. A scalar fitness can't distinguish them. But if you pick A as parent and mutate it, you're propagating "memorize the training set" as a strategy. The next batch is full of overfitters.

So fitness needs to be a vector, and we need a principled way to compare vectors. We compose four mechanisms, in this order:

   raw fitness vector
   (balanced_acc, generalization_gap, param_count, ECE)
                |
                v
   +-----------------------------+
   | 5.1 Pareto NSGA-II          |   non-dominated sort + crowding
   +--------------+--------------+   prevents single-axis gaming
                  |
                  v
   +-----------------------------+
   | 5.2 Novelty (Lehman+Stanley)|   score = a*acc + (1-a)*novelty
   +--------------+--------------+   novelty = kNN dist in
                  |                  confusion-matrix space
                  v
   +-----------------------------+
   | 5.3 ECE-weighted accuracy   |   score *= (1 - lambda*ECE)
   +--------------+--------------+   penalize confident-wrong
                  |
                  v
   +-----------------------------+
   | 5.4 Failure-aware boost     |   if recent fitness delta < 0,
   +--------------+--------------+   raise exploration knobs
                  |
                  v
       final composite score

Figure 4. The scoring stack. Each box takes the previous box's output and adds one more thing the loop should care about.

5.1 Pareto NSGA-II: anti-gaming

The fitness vector is (balanced_accuracy, generalization_gap = val - train, parameter_count, calibration_ECE). We want to maximize the first and minimize the other three.

Pareto sorting is the right tool here. A program dominates another if it's at least as good on every axis and strictly better on at least one. Non-dominated sorting (Deb 2002, NSGA-II) ranks the population into "frontiers": the first frontier contains all non-dominated programs, the second contains all programs dominated only by the first, and so on.

Tournament selection then picks from the lowest frontier (best). Within a frontier, ties break on crowding distance: prefer the program with more empty fitness-space around it, to keep the frontier spread out rather than clumped.

The effect: the LLM can't game a single axis. If it tries to push accuracy up by adding parameters, the param_count axis dominates it back. If it tries to push the generalization gap down by underfitting, accuracy dominates it back. The Pareto frontier is the loop's notion of "good in multiple senses simultaneously."

5.2 Novelty (Lehman & Stanley 2011)

Pareto sorting prevents gaming, but it doesn't drive exploration. A population can be Pareto-diverse while still doing the same thing in different ways. We need an explicit push toward novelty.

The clean formulation comes from Lehman & Stanley's "novelty search" paper: define a behavioral characterization for each program — some descriptor of what it does, not how it scores — and reward programs that are far from their nearest neighbors in that descriptor space.

Our behavioral characterization is the confusion matrix. Two programs have different behavior if they make different kinds of mistakes. Specifically: novelty of child c = mean k-NN distance from c's flattened confusion matrix to the nearest k confusion matrices in the population.

The final novelty-augmented score is:

$$\text{score}(c) = \alpha \cdot \text{accuracy}(c) + (1 - \alpha) \cdot \text{novelty}(c)$$

$\alpha$ is part of the meta-stochastic state and drifts. When the population converges (all confusion matrices look alike), novelty rewards rise, and $\alpha$ can be pushed lower to favor exploration. When the population is spread out, $\alpha$ rises back and accuracy dominates again. We'll come back to this drift in Section 8.

5.3 ECE-weighted accuracy: calibration matters

A model that predicts the right class with 99% confidence and is wrong is worse than a model that predicts the right class with 51% confidence and is wrong. Both lose a point. But the first model is confidently wrong; the second is uncertainly wrong. Soft predictions from the second can still be aggregated usefully in an ensemble; soft predictions from the first poison every ensemble that includes it.

So we penalize miscalibration. Expected Calibration Error (ECE) bins the predictions by confidence and measures the average gap between bin confidence and bin accuracy. We apply it as a multiplicative penalty:

$$\text{score}(c) := \text{score}(c) \cdot (1 - \lambda \cdot \text{ECE}(c))$$

$\lambda$ is in the framework genome, so Level 2 can tune it. Default $\lambda = 0.5$. The penalty is bounded (ECE is in [0, 1], so the multiplier is in [0.5, 1] at $\lambda = 0.5$).

5.4 Failure-aware exploration boost

The last stage is a feedback loop: if the loop is making no progress, open up exploration. We track a rolling window of the last N children's fitness delta (each child's score minus the mean of its parent island). When that rolling mean drops below threshold — meaning the recent batch has been losing ground — the loop responds:

  • Prompt temperature rises. The mutation operator (this Claude Code session) is instructed to produce more aggressive structural mutations rather than safe variants.
  • Novelty alpha drops. The novelty term dominates the accuracy term in the composite score, so the loop rewards being-different over being-good.
  • Tabu strictness rises. Near-duplicate fingerprints get rejected even more aggressively.

When fitness deltas come back positive, all three knobs relax back toward defaults. This is the master valve for exploration; we'll see in Section 7 that there's a per-island cousin of this that fires locally.

Why these four, in this order

Pareto comes first because it operates on the raw fitness vector and is the only stage that produces a partial order (rather than a scalar). Novelty comes next because it transforms a Pareto-best set into a scored set within that frontier. ECE applies multiplicatively after both, because it doesn't care which axis you scored well on. Failure-aware boost is last because it operates over time (rolling window of recent scores), not on a single program in isolation. Each layer reads the output of the previous and adds one more thing the loop cares about.

6. Level 1, part C: constraints

Some children should never be evaluated at all. The challenge has hard rules: no external data, no pretrained weights, no foundation models. If the LLM proposes a program that imports transformers.from_pretrained, shipping it to the cluster is wasted budget. It should be killed in-flight.

Vanilla FunSearch's only constraint is "the code must compile and run." We need more. Three filters in a hard-reject pipeline, plus a per-batch quota:

   child spec
       |
       v
   +-----------------------+   rejection logged to constraint_events
   | 6.1 Rule guards       | --reject--> regenerate with feedback
   +-----------+-----------+   in the mutation prompt
        accept |
        v
   +-----------------------+
   | 6.2 AST tabu          | --reject--> regenerate
   +-----------+-----------+
        accept |
        v
   +-----------------------+
   | 6.3 Lineage cap       | --reject--> regenerate
   +-----------+-----------+
        accept |
        v
   batch assembly + 6.4 family quota (per-batch gate)
        |
        v
   render -> cluster

Figure 5. The constraint pipeline. Three per-child filters, one per-batch quota.

6.1 Rule guards

Hard-coded challenge rules. Auto-reject programs that:

  • import pretrained weight loaders: torch.hub.load, transformers.from_pretrained, huggingface_hub, equivalents
  • access external data: urlopen, requests, wget, gdown
  • exceed parameter cap (default 10M, in framework genome)
  • exceed train-time cap (default 30 min on H200, in framework genome)

Every rejection is logged in constraint_events with the offending rule name. Level 2 (Section 9) reads this table to compute rejection rates per rule per window, and proposes parameter scales when one rule is choking the search.

6.2 AST tabu

Each program is fingerprinted by an AST-based hash that ignores variable names and literal values, but captures structure: which family, which encoder type, which fusion, which loss, which layer arrangement. The last K accepted fingerprints (default K = 50, in framework genome) form a tabu list. Near-duplicate fingerprints are rejected.

This is a cheap exploration prior: you don't need to retrain a near-duplicate of something you trained 20 steps ago. The mutation operator gets the rejection feedback in its next prompt context and learns to propose something structurally different.

6.3 Lineage inbreeding cap

If you keep selecting the same parent and producing children from it, the loop collapses into a single descendant tree. The lineage cap rejects children that trace ancestry to more than N (default 5) consecutive same-parent generations. Combined with tabu, this gives the search a recency-and-relatedness window that prevents the population from inbreeding.

6.4 Family quota (per-batch hard gate)

Filters 6.1–6.3 are per-child. They catch one bad child at a time. But the population can still drift into monoculture through entirely-valid children: every child is structurally novel relative to the last 50, every child respects lineage, but every child is in the same family. We learned this empirically: iter_0012–0014 produced 24 consecutive children from the multi_stream_bigru family, and the loop ground to a halt inside that family's local geometry.

The fix is a per-batch gate, run after children are generated but before the manifest is shipped to the cluster:

  • max_per_family (default 3, range [2, 8]): no model family may exceed this count in one batch of 8.
  • min_families (default 4, range [2, 8]): the batch must span at least this many distinct families.

framework.iteration.allocate_family_slots assigns target families to the N batch slots before mutation prompts are written, under-weighting families that dominated recent batches. validate_batch_family_quota is a hard gate run on the manifest: a batch that violates the quota does not reach the cluster.

There's also a blender rule. If a family has 2 or more slots in a batch, the second slot's child must differ from the first via a structural operator (swap_encoder, swap_fusion, etc.), not a hyperparameter knob. assign_blender_flags tags each slot with the operator type the mutation operator is required to use.

A note on what we removed

A previous "curriculum_unlock" filter gated complex families until simpler ones had been explored first. It was scorched on 2026-05-11 (logged in CLAUDE.md). Threshold-based gating proved brittle on this dataset's hard ceiling: simple families never crossed the threshold because the ceiling itself was at ~0.53. All families with entry points in FAMILY_ENTRY_POINTS are eligible from iteration 1. The Pareto param_count axis handles the simplicity-vs-complexity tradeoff cleanly.

7. Level 1, part D: breakdown mechanisms

Stuck mechanisms need unsticking. Vanilla FunSearch only resets the bottom-K islands every R iterations. That's a blunt tool. We layer three more focused mechanisms on top:

  stagnation signals
        |
        +--> 7.1 Inter-island migration
        |       trigger: island gap >= migration_patience
        |       action: child is composed from THIS island's parent
        |               and ANOTHER island's parent (crossover)
        |
        +--> 7.2 Coevolutionary critic (Hillis 1990)
        |       a parallel evolving population of HARD test cases
        |       evolves to maximize program failure rate
        |       programs evolve to survive critics
        |       an actual arms race
        |
        +--> 7.3 Per-island stagnation -> escalating temperature
                trigger: island best hasn't improved in K generations
                action: raise THIS island's prompt temperature
                        + novelty alpha, force structural mutations

Figure 6. Three breakdown mechanisms. They fire independently and target different scales: 7.1 cross-island, 7.2 against the val set's hardness, 7.3 within one island.

7.1 Inter-island migration

If an island's best fitness flatlines for K generations (default K = 10, in genome), its next child gets a foreign parent. The mutation operator's prompt now contains two parent specs — one from this island, one from a stuck-resistant peer island — and is asked to compose a crossover child from both. This is genuine cross-pollination, not just within-island variation.

The wiring lives in framework.iteration.prepare_batch(migration_patience=...): when the trigger fires, the per-slot prompt includes both parent_spec and foreign_parent_spec, and the mutation operator emits a hybrid.

7.2 Coevolutionary critic (Hillis 1990)

Most evolutionary search uses a fixed fitness landscape: the val set is the val set, programs evolve to climb it, and the val set never fights back. Hillis's insight is that you can make the landscape evolve too. Run a second population whose individuals are hard test cases — subsets of the val set, signal perturbations, channel permutations. Critics evolve to maximize program failure rate. Programs evolve to survive critics. An arms race emerges, and both populations push each other into higher fitness.

In our framework, a critic individual is a triple:

  • subject_subset: which val subjects to score on (a hard subset is one most programs do poorly on)
  • signal_perturbation: noise / occlusion / time-warp applied to inputs
  • channel_permutation: scramble or zero-out a subset of the 4 channels

The critic population has its own ledger table (critic_population), its own GENITOR replacement, and its own fitness: a critic's fitness is the average failure rate of the current program population on that critic. Hard critics survive; easy ones get replaced.

Each batch, one new critic is evolved (currently by mutating channel_permutation) and the top critics are surfaced in the mutation prompt via include_critic_in_prompt. The full critic-aware training fitness (training programs to be robust to the critic population in their loss) is deferred to a cluster-side sprint — we wired the critic population, but the program fitness function still uses the standard val set. The critic infrastructure is there; the next iteration of the framework will close the loop.

7.3 Per-island stagnation → escalating temperature

The smallest-scale unsticker: if a single island's best hasn't improved in K generations, that island's local meta-stochastic state escalates. The prompt temperature for that island multiplies by 1.5. The novelty alpha increases by 0.2. Forced structural-mutation operators get higher prior in the mutation prompt.

This is the local cousin of the global failure-aware boost in Section 5.4. The global version pushes the whole loop. The local version pushes a single island. They coexist because they target different signals: global stagnation is "nothing in the population is improving"; local stagnation is "this corner of the population is dead."

8. Level 1, part E: the meta-stochastic layer

So far Level 1's parameters have been "defaults plus failure-aware boost when stuck." We can do better. Two parameters drift stochastically across iterations, persisted in the meta_state table:

8.1 Mix-ratio drift

The mutation prompt has two modes, biased by a single number p_lit in [0.2, 0.8]:

  • When p_lit is high, the prompt biases toward literature: "what would the most-cited paper on this signal type do? What's the standard inductive bias for periodic physiological signals?"
  • When p_lit is low, the prompt biases toward novelty: "what is a novel structural mutation that would surprise me? What's a combination of operators that doesn't appear in the recent population?"

p_lit does a Gaussian random walk: every 10 iterations, draw a step from $\mathcal{N}(0, \sigma^2)$ with default $\sigma = 0.05$, then clamp to [0.2, 0.8]. Over the course of a long run, the loop spends time in literature-mode and time in novelty-mode without ever committing to either.

Why a random walk? Because the optimal mode is itself unknown and time-varying. Early in the search, literature-mode is high-value (you don't know what the basic strong baselines look like, the literature has them). Late in the search, when the basic baselines are exhausted, novelty-mode is high-value (you need to find combinations that haven't been published). A random walk lets the loop sample both regimes naturally.

8.2 Failure-aware boost (the master valve)

Already described in Section 5.4. The recap, in this context: when recent fitness deltas turn negative, this single mechanism opens up every exploration knob simultaneously: prompt temperature rises, novelty alpha rises, tabu strictness rises, lineage cap tightens. When fitness recovers, the valve closes back toward defaults.

In ledger terms: the failure-aware boost reads median_fitness_delta_per_island(window) and writes back to the meta_state row at the end of each batch. The next batch's mutation prompts pick up the updated knobs automatically.

The wiring

As of 2026-05-11, both 8.1 and 8.2 are live: each batch, prepare_batch(evolve_meta=True) loads the latest meta_state row, calls step_meta_state (which drifts p_lit + applies update_failure_boost + relax_toward_defaults), and persists the new state. Across batches, the meta-stochastic knobs really do evolve. Level 2 can later mutate the drift schedules themselves.

9. Level 2: the framework mutates itself

Now we get to the part that makes this a three-level framework and not just a fancy evolutionary algorithm.

Everything in Sections 4–8 was Level 1: an engine with fixed structure and a few knobs that drift. The engine can get stuck. When it gets stuck, the meta-stochastic layer (Section 8) opens up exploration. That handles most stagnation.

But what if the engine is stuck because Level 1's structure is wrong for this problem?

Concretely: what if the AST tabu length is too aggressive and is rejecting 80% of children? The failure-aware boost will fire, exploration will open up, but the tabu list is still rejecting most of what the loop produces. The knob you need to turn is the tabu length itself, not the prompt temperature.

Or: what if a particular family has dominated the population for 24 children, and the family quota max_per_family = 3 is too lax to catch it? You need to tighten the quota, not just push novelty.

Or: what if a single island is consistently the worst and never recovers? You need to drop the island, not just reset its bottom-K.

These are all structural changes to Level 1. They can't be expressed as drift on existing knobs. So we need a layer that can propose them, validate them, and apply them with appropriate guardrails. That layer is Level 2.

9.1 The framework genome

Level 2 works on the framework genome: a JSON snapshot of every Level 1 parameter and operator choice. The schema lives in framework/introspect.py:GENOME_RULE_GUARDS. A small slice of it:

{
  "island_count": 4,                // range [2, 8]
  "island_size": 8,                 // range [4, 16]
  "reset_cadence": 50,              // range [10, 200]
  "novelty_alpha": 0.7,             // range [0.3, 0.95]
  "ece_lambda": 0.5,                // range [0.0, 1.0]
  "ast_tabu_k": 50,                 // range [10, 200]
  "lineage_cap": 5,                 // range [2, 10]
  "max_per_family": 3,              // range [2, 8]
  "min_families": 4,                // range [2, 8]
  "migration_patience": 10,         // range [5, 30]
  "p_lit_drift_sigma": 0.05,        // range [0.01, 0.20]
  "failure_boost_gain": 1.5,        // range [1.0, 3.0]
  "replacement_rule": "GENITOR",    // enum {GENITOR, TOURNAMENT}
  ...
}

Every field has a declared range (the "rule guard"). A Level 2 mutation that proposes a value outside the range is rejected before it ever reaches the live state. This is the safety contract: Level 2 can tune Level 1, it cannot break Level 1.

9.2 When does Level 2 fire?

You might be expecting "every M iterations." That was the placeholder when we first sketched the framework. It's the wrong answer. M is arbitrary; firing on a fixed cadence will sometimes fire when nothing is wrong, and other times will silently miss real pathologies.

The right trigger is compound, empirical, and read from the ledger. Level 2 fires when ANY of three pathologies appears in recent ledger state, AND a few guards are satisfied:

   end of each iteration:

   +--------------+--------------+

Figure 7. The compound detector. Three pathologies, three guards: bootstrap floor (don't fire on a cold start), chatter guard (don't fire twice in 10 iterations), silence cap (don't go more than 100 iterations without checking in).

The thresholds (window size 20, stagnation epsilon 0.005, rejection cutoff 0.6, entropy ratio 0.7, min/max gap 10/100) all live in the genome. They can themselves be mutated by future Level 2 fires — the introspection layer can adjust its own sensitivity.

9.3 The seven typed operators

Level 2 doesn't propose arbitrary diffs. It picks from a small typed catalog of mutations:

OpSchemaUse when
ADD_ISLAND(init_genome_perturbation)Diversity collapse and island count below cap.
DROP_ISLAND(island_id)Persistent worst island, island count above floor.
SCALE_PARAM(field, factor in [0.5, 2.0])Tune one numeric knob inside its range.
TOGGLE_OPERATOR(name, on/off)Enable or disable a Level 1 component (e.g. failure_boost).
SWAP_OPERATOR(slot, name)Replace one operator with a sibling (e.g. GENITOR -> TOURNAMENT).
SET_THRESHOLD(name, value)Adjust stagnation, migration, or lineage threshold.
SET_AXIS_WEIGHT(axis, weight)Re-weight a Pareto axis.

Why typed and not freeform? Because freeform diffs are unverifiable. A typed operator has a known schema and a known safety contract. SCALE_PARAM(ast_tabu_k, 0.5) is interpretable, range-checkable, and reversible. "Rewrite the constraint pipeline" is none of those things.

9.4 The proposal priority chain

When the compound detector fires, the proposal function reads the ledger and selects an operator by priority:

  1. Family monoculture dominant → SET_THRESHOLD(max_per_family, 2). Fires when detect_family_monoculture sees a single model family at ≥70% of the last 24 children. This is the highest priority because a same-family monoculture means architectural search has stopped entirely — which strictly dominates the knob-level pathologies below.
  2. Over-rejection dominant → tweak the offending constraint. Example: tabu rejected >60% → SCALE_PARAM(ast_tabu_k, 0.5) (shorten the tabu list).
  3. Stagnation dominant → SCALE_PARAM(novelty_alpha, 1.5) or SCALE_PARAM(p_lit_drift_sigma, 1.5) or SWAP_OPERATOR(replacement_rule, "TOURNAMENT").
  4. Diversity collapse dominant → SET_AXIS_WEIGHT(novelty, +) or ADD_ISLAND.

An approved proposal goes through one more check before it's applied: the post-mutation genome must still satisfy GENOME_RULE_GUARDS, and it cannot disable any Level 1 hard constraint. (This is anti-pattern 13 in ANTIPATTERNS.md: Level 2 can tune Level 1 parameters within declared ranges, but it cannot redefine the constraint set itself.)

If the check passes and the human gate (HIP-H) confirms, the mutation is applied to live state, persisted in framework_mutations with a timestamped row, and the next batch's mutation prompts read the new genome automatically.

The lineage

Level 2 is the layer with the strongest published roots: Schmidhuber's Gödel Machines (a self-rewriting reinforcement learner with proof-based safety), DeepMind's AlphaEvolve (2025; LLM-driven program evolution with structural diffs), and Wang et al.'s POET (2019; co-evolution of environments and agents). All three are precedents for "the optimizer changes its own structure under observation." Our typed-operator catalog and rule-guard contract are the safety layer that makes this practical on a human-timescale loop.

10. The workflow: git as the only bridge

Up to this point the framework has been described as if it runs on one machine. It doesn't. The mutation operator (this Claude Code session) runs on Vignan's Mac. Training runs on the NEU Explorer cluster (1 H200 per task, 8 hour limit, 8 concurrent jobs, 12 GB RAM). The Mac and the cluster need a way to talk to each other.

The temptation is to script that communication: have the Mac ssh into the cluster, rsync files, sbatch jobs, and watch for results. That's the "natural" engineering instinct. We don't do any of it.

Git is the only bridge. Mac and cluster communicate exclusively through git push and git pull. No SSH from the Mac, no scp, no paramiko, no inline-JSON sbatch --wrap. The framework Python on the Mac never touches the cluster. Every git push, every ssh, every sbatch, every git pull is a manual command Vignan types.

Why this hard constraint?

  1. Reproducibility. Every state transition lives in git. If you want to know what happened in iter_0017, you check out that commit. There's no "what was on the cluster's disk at the time" question to answer.
  2. Cognitive load. Constructing forensic SSH commands per iteration ("ssh in, cat this file, copy-paste the output back") is slow and error-prone. With git as the bridge, the per-iteration command list is the same four shell commands, always.
  3. Blast radius. A Mac-side script that does ssh ... rm ... can wipe a cluster directory by accident. A Mac-side script that can't SSH literally cannot do that. The system is safer because the operator surface is smaller.
  4. State divergence. "What got pasted into chat" and "what's on disk" can drift. With git, there is no chat-state. All cluster state lives in git, where reproducibility lives.

The four-command chain

One batch of the loop is exactly four operator actions (plus the Mac-side Claude reasoning that produces the specs):

  Mac (Claude session, no shell)
    -> generate experiments/iter_NNNN/child_MM/{spec.json, run.py}
       and manifest.json
    -> write ledger rows (experiments, mutation_traces,
       constraint_events)
    -> /Commit-Initiation: plan the commit, do not execute

  Mac (Vignan, shell)              # command 1
    -> git commit + git push

  Cluster (Vignan, shell)          # commands 2 + 3
    -> git pull
    -> sbatch --array=0-(N-1)%4 \
              --export=ALL,MANIFEST=experiments/iter_NNNN/manifest.json \
              scripts/run_array.slurm
       (cluster trains N children in parallel, writes result.json
        into each child_MM/)

  Cluster (Vignan, shell)          # command 4 (after training)
    -> git add experiments/iter_NNNN/child_*/result.json
       git commit + git push

  Mac (Vignan, shell)              # command 5
    -> git pull

  Mac (Claude session)
    -> read result.json files from disk
    -> write fitness vectors to ledger
    -> update meta-stochastic state, evolve critic, check_level2
    -> reason next batch

Figure 8. One batch. Five manual commands across two machines, no automation, no SSH from the Mac.

What's tracked in git, what isn't

Two principles:

  • result.json and trace.jsonl ARE tracked. These are small text files. They are the canonical record of what each child did, and they MUST round-trip through git so the Mac sees the same artifact the cluster wrote.
  • checkpoints/ is NOT tracked. Model weights are large binaries. They live on the cluster filesystem; the Mac reads probability tables from the per-seed JSONs that are tracked. The submission pipeline preserves predictions, not weights.

Anti-pattern 11a (in ANTIPATTERNS.md) formalizes the git-only contract. If you ever feel the urge to write a Mac-side script that touches the cluster, stop and re-read 11a.

11. Observability: how Level 2 sees what's happening

Level 2 can only make meaningful proposals if it can see why Level 1 is stuck, not just that it's stuck. Three observability surfaces give it that signal.

11.1 mutation_traces + JSONL mirror

Every accepted child is logged twice, atomically:

  • A row in mutation_traces (SQLite): iteration, run_id, parent_run_ids, prompt_context, child_spec, fingerprint, reasoning_summary, accepted, created_at.
  • An appended line to experiments/<run_id>/trace.jsonl with the same payload, JSONL-encoded.

Why both? The JSONL is the durable write-ahead log: append-only, replay-safe, greppable from the cluster side. The SQLite table is the queryable materialized view that Level 2's detectors run against. If the SQLite ever corrupts, you can replay from the JSONL. Cluster runs append JSONL directly; the Mac queries via SQLite.

The reasoning_summary field is the 2–5 sentence rationale this Claude Code session emits as a top-level field in every child spec. It's the human-readable answer to "why this child?" When Level 2 proposes a mutation, the trace history of recent children gives it the LLM-written narrative of how the population got where it is.

11.2 constraint_events

Every rule check — rule_guard, AST tabu, lineage cap, family quota — logs a row to constraint_events: which rule, accepted or rejected, reason code, child run_id. This is what lets Level 2 compute per-rule rejection rates per window. Without it, "the loop is over-rejecting" would be an unanswerable question.

11.3 Query helpers on Ledger

The ledger exposes a small set of pre-computed signals to keep Level 2's logic simple:

  • median_fitness_delta_per_island(window) → stagnation-detector input
  • constraint_rejection_rate(window, rule=None) → over-rejection-detector input
  • fingerprint_entropy(window) → diversity-collapse-detector input
  • recent_mutation_traces(window) → proposal context for the LLM

All four are O(window) SQL queries. Level 2's compound detector composes them; the priority chain reads them to pick an operator.

Why no LangSmith / external observability

Adding an external observability service would add an external LLM dependency, which violates anti-pattern 11 (no external LLM SDKs). It would also fragment the trace surface: some history in the local ledger, some in the cloud. We keep everything local: SQLite for queries, JSONL for replay, both in the repo, both versioned with the code. If you ever need to debug iter_0017, you check out the commit and you have the full state, no service to log into.

Closing

That's the framework. Three levels, ten mechanisms, four ledger tables, one git-as-bridge contract. The whole thing was designed around a single empirical thesis: the AI4Pain 2026 benchmark is asking for cortical information from a peripheral channel set, and the only path to a competitive result is to search hard with every diversity-preserving mechanism we can justify from the literature.

The architectural choices have lineage. FunSearch (Romera-Paredes et al., Nature 2024) gives us the island structure. GENITOR (Whitley 1988) gives us steady-state replacement. NSGA-II (Deb 2002) gives us multi-objective Pareto. Novelty search (Lehman & Stanley 2011) gives us behavioral exploration. Hillis (1990) gives us coevolution. Promptbreeder (Fernando et al. 2023) gives us prompt-mode drift. Schmidhuber Gödel Machines, AlphaEvolve (DeepMind 2025), and POET (Wang et al. 2019) give us self-modification under safety constraints. Each piece is there because it answers a concrete pain point in the wish list at the start of Section 1.

The framework is the contribution. AI4Pain 2026 is the first case study; the empirical results live in SUBMISSIONS.md, and the per-architecture details of what landed in the 5 portal submissions live in MODELS_FRAMEWORK.html. Future case studies will deploy the same engine on new domains. This document is the methodology snapshot used here.

If you found a confusing section, an unmotivated jump, or a section that should have its own subsection, the doc is meant to be edited. The goal is a mental model crisp enough to write a workshop paper from. That's the test.
